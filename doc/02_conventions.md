# Conventions

## Границы изменений

Все изменения в проекте должны быть минимальными и понятными.  
На каждом этапе разработки нужно придерживаться принципов минимального diff и KISS (Keep It Simple, Stupid): не добавлять лишние функции, не усложнять архитектуру без необходимости и не менять сразу несколько частей системы, если задача касается только одной.

Каждая новая итерация должна решать одну конкретную задачу: например, загрузка данных, построение индекса, улучшение retrieval или добавление UI.  
Если изменение нельзя быстро объяснить в 2–3 предложениях, значит оно, скорее всего, слишком большое для MVP.

## Архитектура модулей

Проект должен быть разделён на простые и логичные модули.  
Базовая структура MVP:

- `ingest` - загрузка и чтение исходных данных;
- `chunk` - преобразование данных в текстовые фрагменты;
- `index` - построение и сохранение эмбеддингов и векторного индекса;
- `retrieval` - поиск релевантных фрагментов по запросу пользователя;
- `ui` - пользовательский интерфейс (консольный или web);
- `main` или `app` - точка входа для запуска проекта.

Каждый модуль должен решать только одну основную задачу.  
Нельзя смешивать в одном файле сразу загрузку данных, построение индекса, retrieval и UI, если это мешает читаемости и тестируемости.

## Правила ответа

RAG-система должна отвечать только на основе найденного контекста.  
Если в найденных данных нет достаточной информации для ответа, система должна честно сообщать, что не может ответить на вопрос на основе доступных данных.

Основные правила ответа:
- не придумывать факты, которых нет в контексте;
- не использовать внешние знания модели как основной источник ответа;
- если релевантные данные не найдены, возвращать вежливый отказ;
- если контекст частичный, явно указывать, что ответ ограничен найденными данными;
- по возможности делать ответ кратким, понятным и связанным с конкретными фрагментами данных.

Пример корректного отказа:
> Я не могу надёжно ответить на этот вопрос, потому что в найденных данных нет достаточной информации.

## Тестовые требования

Для каждой новой версии проекта нужно проверять не только запуск кода, но и поведение RAG-системы.  
Минимальный набор тестов для MVP:

- проект запускается локально без ошибок;
- данные успешно загружаются;
- индекс строится и сохраняется;
- retrieval возвращает релевантные фрагменты по тестовым вопросам;
- система отвечает осмысленно, если контекст найден;
- система отказывается отвечать, если контекст отсутствует или недостаточен.

Также желательно иметь набор из нескольких тестовых вопросов:
- вопрос, на который ответ есть в данных;
- вопрос, на который ответ частично есть в данных;
- вопрос, на который ответа в данных нет.

## Правила по данным и зависимостям

В MVP используется один основной источник данных, чтобы не усложнять систему.  
Если данные табличные, они должны быть преобразованы в текстовый формат, пригодный для индексации и retrieval.

Правила работы с данными:
- не изменять исходный датасет вручную без явного описания причины;
- все этапы преобразования данных должны быть воспроизводимыми;
- данные для индекса должны строиться из исходного источника автоматически или полуавтоматически;
- если добавляются новые данные, это должно быть отражено в документации проекта.

Правила по зависимостям:
- использовать минимально необходимый набор библиотек;
- не добавлять тяжёлые зависимости без необходимости;
- все зависимости должны быть зафиксированы в `requirements.txt`;
- версии библиотек должны быть совместимыми и достаточными для локального запуска.

## Как проверить

Файл `conventions.md` считается корректным, если по нему можно принимать решения на всех следующих этапах проекта.  
Проверка заключается в том, что для любой новой задачи можно ответить на вопросы:
- изменение маленькое и понятное?
- оно укладывается в MVP?
- оно не ломает архитектуру модулей?
- система по-прежнему отвечает только по контексту?
- данные и зависимости остаются воспроизводимыми?

Если на эти вопросы можно ответить “да”, значит правила сформулированы достаточно хорошо и подходят для дальнейшей разработки.
