# Vision

## Технологии

Для MVP проекта будет использоваться Python как основной язык разработки.  
Для обработки данных планируется использовать pandas, для подготовки текстов - стандартные инструменты Python, для retrieval - эмбеддинги и векторный индекс FAISS, а для генерации ответов - LLM через API.

В качестве пользовательского интерфейса для MVP будет использоваться простой web-интерфейс, например Streamlit, либо консольный интерфейс на первом этапе.  
Главная цель MVP - не сделать сложное приложение, а собрать работающий end-to-end прототип RAG-системы.

## Как строится индекс

Источник данных для проекта - табличный датасет с Kaggle **Social Media Impact on Teen Mental Health**.  
Так как данные в датасете представлены в строках таблицы, на этапе подготовки каждая запись будет преобразована в текстовое описание, включающее основные признаки: использование социальных сетей, сон, стресс, тревожность и другие характеристики.

После этого текстовые описания будут разбиваться на фрагменты при необходимости, преобразовываться в эмбеддинги и сохраняться в векторный индекс FAISS.  
Таким образом, индекс будет состоять не из исходных числовых строк, а из текстовых представлений записей, пригодных для семантического поиска.

## Как работает поиск

Когда пользователь задаёт вопрос, этот вопрос преобразуется в embedding с помощью той же модели эмбеддингов, которая использовалась для индексации данных.  
Далее выполняется similarity search по векторному индексу, и система находит несколько наиболее релевантных текстовых фрагментов.

Найденные фрагменты передаются в LLM как контекст вместе с вопросом пользователя.  
На основе этого контекста модель формирует ответ, который должен быть связан с данными датасета, а не только с общими знаниями модели.

## Что не используем в MVP

В MVP не планируется использовать:
- сложную многошаговую агентную логику;
- переоценку и reranking документов отдельной моделью;
- несколько источников данных одновременно;
- сложную авторизацию и многопользовательский режим;
- продвинутый frontend;
- дообучение собственной языковой модели.

Также в MVP не будет полноценной аналитической платформы с графиками, фильтрами и дашбордами.  
Основной фокус - построить минимально работающий RAG-пайплайн: загрузка данных, преобразование в текст, индексация, retrieval и генерация ответа.

## Ограничения MVP

У проекта есть несколько ограничений:
- используется только один основной датасет;
- качество ответа зависит от качества преобразования табличных строк в текст;
- система не является медицинским инструментом и не должна использоваться для постановки диагнозов;
- ответы могут быть ограничены содержимым выбранного датасета и не покрывают всю тему подросткового ментального здоровья.

Кроме того, MVP ориентирован на демонстрацию подхода RAG, а не на production-ready решение.  
Если окажется, что табличного датасета недостаточно для качественного retrieval, на следующих этапах проект может быть расширен дополнительными текстовыми источниками.

## Как запускать проект

Предполагается, что проект будет запускаться локально.  
Базовый сценарий запуска:
1. Установить зависимости из `requirements.txt`.
2. Подготовить и обработать датасет.
3. Построить или загрузить индекс.
4. Запустить приложение или скрипт с интерфейсом вопросов и ответов.

Если для генерации ответов используется внешний API, перед запуском также потребуется указать ключ доступа в переменных окружения.

## Как проверить

Проверка MVP будет выполняться с помощью списка тестовых вопросов по датасету.  
Например:
- Какие факторы чаще связаны с высоким уровнем стресса?
- Есть ли связь между временем в социальных сетях и качеством сна?
- Какие характеристики чаще встречаются у подростков с высоким уровнем тревожности?

MVP считается рабочим, если:
- проект запускается локально без сложной настройки;
- индекс строится корректно;
- по пользовательскому вопросу находятся релевантные записи;
- итоговый ответ использует найденный контекст и логически связан с данными.
