# Workflow

## Цикл одной итерации

Каждая итерация проекта проходит по одному и тому же циклу:
1. Понять цель текущей итерации.
2. Зафиксировать, что именно должно быть сделано.
3. Выполнить только те изменения, которые нужны для этой итерации.
4. Проверить результат.
5. Зафиксировать изменения в репозитории.
6. Обновить статус в `doc/03_tasklist.md`.
7. Только после этого переходить к следующей итерации.

Одна итерация должна завершаться маленьким, но проверяемым результатом.  
Если задача слишком большая и её трудно проверить, её нужно разделить на более мелкие шаги.

## Что делать до кода

Перед началом любой итерации нужно:
- открыть `doc/03_tasklist.md` и выбрать текущую итерацию;
- прочитать связанные ограничения из `doc/01_vision.md`;
- проверить правила из `doc/02_conventions.md`;
- зафиксировать, что является минимальным результатом текущего шага;
- понять, как именно будет выполняться проверка.

До написания кода должно быть ясно:
- какой файл или модуль изменяется;
- что именно должно заработать после изменения;
- как понять, что результат корректный.

Если это неясно, начинать кодить нельзя.

## Как выполнять проверку

Проверка выполняется в конце каждой итерации, а не только в конце проекта.  
Для каждой итерации должна быть простая и понятная команда проверки, например:
- скрипт запускается без ошибок;
- данные читаются корректно;
- индекс строится;
- retrieval возвращает результаты;
- ответ генерируется по контексту;
- UI открывается и принимает вопрос.

Проверка должна быть воспроизводимой.  
Это значит, что другой человек должен суметь повторить шаги проверки по описанию и получить тот же результат.

## Когда обновлять tasklist

`doc/tasklist.md` нужно обновлять после завершения каждой итерации.  
После проверки результата необходимо:
- отметить текущую итерацию как завершённую или обновить её статус;
- при необходимости уточнить следующую итерацию;
- зафиксировать изменения в проверке или scope, если они появились по ходу работы.

Tasklist нельзя держать “на потом”.  
Он должен отражать текущее состояние проекта после каждого завершённого шага.

## Правила коммита и перехода дальше

Коммит нужно делать только после того, как текущая итерация проверена.  
Каждый коммит должен быть атомарным, то есть содержать одно логически цельное изменение.

Правила коммита:
- один коммит — одна завершённая маленькая задача;
- сообщение коммита должно коротко объяснять суть изменения;
- не смешивать документацию, рефакторинг и новую функциональность без необходимости;
- не переходить к следующей итерации, пока текущая не проходит проверку.

Примеры хороших commit message:
- `add dataset loader`
- `build text transform for records`
- `add faiss indexing`
- `update tasklist after retrieval iteration`

Переходить дальше можно только тогда, когда:
- код или документ сохранён;
- проверка выполнена;
- результат зафиксирован в репозитории;
- tasklist обновлён.

## Запрещённые действия

В рамках MVP запрещается:
- перепрыгивать через итерации без завершения предыдущих;
- одновременно менять слишком много частей проекта;
- добавлять сложные функции вне scope MVP;
- использовать непроверенные зависимости без необходимости;
- оставлять код без проверки;
- делать коммит с незавершённым или неработающим результатом;
- менять архитектуру проекта без обновления `01_vision.md` и `02_conventions.md`;
- заставлять RAG отвечать “из головы”, если контекста недостаточно.

Также запрещено расширять проект до production-уровня раньше времени.  
Сначала должен быть готов простой, понятный и рабочий MVP.

## Как проверить

Файл `04_workflow.md` считается корректным, если по нему можно пройти любую итерацию без неоднозначности.  
Проверяющий должен понимать:
- что нужно сделать до начала работы;
- когда писать код;
- как проверить результат;
- когда обновлять tasklist;
- когда делать commit;
- когда можно переходить к следующему шагу.

Если по документу можно воспроизвести одну полную итерацию от планирования до фиксации результата, значит workflow описан достаточно хорошо.
